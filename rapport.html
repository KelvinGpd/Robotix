Cas d'utilisation : Achat de composantes
But : Un utilisateur quelconque du service souhaite se procurer les composantes nécessaires afin de contruire son robot.
Préconditions :
1. L'utilisateur est inscrit et connecté sur le système Robotix
2. L'utilisateur est capable d'effectuer une transaction par carte en ligne
3. Le fournisseur est inscrit au système et peut recevoir une commande

Acteurs : Utilisateur (principal), Fournisseur (secondaire)
Scénario principal : description étape par étape, ordonnée numériquement
1. L'utilisateur se connecte au marché.
 1.1. Il y entre le nom de la composante(ou l'identifiant) qu'il recherche.
2. Le système recherche à travers sa base de données pour les pièces satisfaisant la recherche.
2.1 Il y renvoie a lutilisateur une liste.
3. L'utilisateur ajoute le ou les composantes désirées dans son panier.
3.2. Il procède au paiement sécurisé.
4. Le système reçoit la requête et les détails de la commande.
4.1 le système confirme la commande et l'envoie au fournisseur.
5. le fournisseur complète la commande en envoyant les produits à l'utilisateur.
Scénarios alternatifs : ramifications du déroulement principal, alternatives possibles à certaines étapes
2a. Le système ne trouve aucune composante pouvant satisfaire la recherche de l'utilisateur.
2a.1. Le système renvoie une erreur avertissant qu'il n'y existe aucune pièce sur le marché.
2a.2. Le scénario recommence à l'étape 1.
4.1a. Le système ne peut confirmer la commande du à une erreur de paiement, manque de stock...
4.1a.1 Le système envoie une erreur, et le scénario recommence à l'étape 1.
Postconditions : Le système enregistre les informations suivantes:
-La date et l'heure à laquelle la commande à été envoyée
-La date et l'heure à laquelle la commande fut complétée
-Les informations propre à la commande(composante id, quantité, prix)
-L'identifiant du client ayant passé la commande
-L'identifiant du fournisseur ayant complété la commande

Cas d'utilisation : Lister une composante sur le marché.
But : Un fournisseur quelconque souhaite lister une composante sur le service pour le vendre.
Préconditions :
1. Le fournisseur est inscrit sur le système Robotix
2. Le fournisseur est en mesure de fournir toutes les informations propres à la composante

Acteurs : Fournisser (principal).
Scénario principal : description étape par étape, ordonnée numériquement
1. Le fournisseur se connecte à son compte
 1.1. Il accède à la page de ses produits
1.2. Il sélectionne "rajouter un produit" et entre tous les détails, (Serial number, available stock...)
1.3. Il envoie la demande de lister le nouveau produit au système. 2. Le système reçoit la demande et identifie le fournisseur
2.2 Le sytème rajoute dans sa base de données le produit ainsi que toutes les informations ratachées.
2.3 Le fournisseur reçoit une alerte comme quoi sa composante à été mise sur le marché.

Scénarios alternatifs : ramifications du déroulement principal, alternatives possibles à certaines étapes
1.3a. Les détails fournis sont invalides(vides ou ne suivent pas REGEX)
1.3a.1 Le système renvoie une alerte comme quoi il y a des champs invalides, et le processus reprend à 1.2
Postconditions : État du système lorsque le scénario principal achève avec succès.
Le système enregistre les informations suivantes:
-L'identifiant du fournisseur ratachée à la composante
-Les informations propres à la composante, notamment: Le prix unitaire, La stock disponible, La description ainsi que l'id.

Cas d'utilisation : S'inscrire au système comme utilisateur
But : L'utilisateur s'inscrit au système en saisant les informations nécessaires.
Préconditions :
1. L'utilisateur a une adresse courriel et un numéro de téléphone
2. L'utilisateur travaille pour une compagnie
Acteurs : Utilisateur
Scénario principal :
1. Le système demande à l'utilisateur de saisir les informations nécessaires.
1.1. L'utilisateur saisit les informations.
2. Le système vérifie que les informations sont valides.
2.1. Pseudo est unique?
2.2. L'addresse courriel et le numéro de téléphone sont dans le bon format?
2.3. La compagnie existe?
3. Le système indique à l'utilisateur qu'il/elle a été inscrit avec succès.
Scénarios alternatifs :
2a. Les informations saisies pas l'utilisateur ne sont pas valides.
2a.1. Les informations ne sont pas enregistrés dans le système.
2a.2. Le système avertit l'utilisateur que les informations ne sont pas valides.
2a.3. Le scénario recommence à l'étape 1 Postconditions : Les informations de l'utilisateur sont enregistrées dans le système.

Cas d'utilisation : Créer une action
But : L'utilisateur crée une action pour ses robots.
Préconditions :
1. L'utilisateur est inscrit dans le système Robotix
Acteurs : Utilisateur
Scénario principal :
1. L'utilisateur saisit les instructions pour effectuer une action.
2. Le système vérifie que ces instructions ne contiennent pas d'erreurs et sont possibles à effectuer avec les composantes offerts par Robotix.
3. Les instructions pour cette action sont enregistrées dans le système.
Scénarios alternatifs :
2a. Les instructions sont impossibles à effectuer ou contient des erreurs.
2a.1. Le système indique à l'utilisateur que les instructions ne sont pas bonnes.
2a.2. L'utilisateur corrige les instructions et le système les vérifie une autre fois.
Postconditions : Les instructions pour l'action sont enregistrés dans le système.

Cas d'utilisation : Assigner une série d'actions(une tâche) à un robot
But : Un utilisateur du système souhaite attribuer une tâche à un de ses robots.
Préconditions : préconditions nécessaires avant d'entammer le scénario principal.
1. L'utilisateur est inscrit et sur le système Robotix.
2. L'utilisateur possède au moins un robot dans sa flotte.
3. L'utilisateur à dèjà crée plusieurs actions assignables.

Acteurs : Utilisateur.
Scénario principal : description étape par étape, ordonnée numériquement
1. L'utilisateur rentre dans la page "gérer mes robots"
 1.1. Le système récupère la liste de tous les robots associés à l'identifiant de l'utilisateur (Si le robot n'est pas disponible, le système le saute).
 1.1. L'utilisateur sélectionne le robot dans la liste auquel il désire attribuer une tâche.
2. Le système récupère la liste de tous les actions prédéfinis par l'utilisateur et les affiche.
3. L'utilisateur ordonne une liste séquentielle d'actions à partir du Gui. Optionnellement, il choisit aussi un horaire et/ou une action déclencheur.
3.1 L'utilisateur sélectionne "create & assign task".
3.2 Le système vérifie la validité de la tâche et envoie un message de confirmation à l'utilisateur.
4. Le système attribue la tâche dans sa base de données à l'identifiant du robot.
Scénarios alternatifs : ramifications du déroulement principal, alternatives possibles à certaines étapes
1.1a. tous les robots de l'utilisateur sont occupés
1.1a.1 Le système renvoie une alerte à l'utilisateur comme quoi aucun de ses robots n'est disponible. Le processus se termine (exit process).
3.2a. La liste séquentielle d'actions est invalide, car le robot n'est pas en mesure d'effectuer un ou plusieurs actions dans la liste.
3.2a.1 Le système renvoie une alerte comme quoi; "robot-name" cannot execute "task-name" because it is missing the required components. Le procès recommence à partir de l'étape 3.
3.2b. La liste séquentielle d'actions est invalide, car la liste d'actions spécifée par l'utilisateur est vide.
3.2b.1 Le système renvoie une alerte comme quoi; please select at least one one action to create a task. Le procès recommence à partir de l'étape 3.
Postconditions : État du système lorsque le scénario principal achève avec succès.
Le système enregistre les informations suivantes:
-Les informations principales de la tâche(liste d'actions) et ceux optionnels (horaire et action déclencheur)
-L'identifiant du robot dans la flotte de l'utilisateur ayant été attribué la tâche.

Cas d'utilisation : Voir l'état de ses robots
But : L'utilisateur voit l'état de ses robots tel que la position, condition, vitesse, etc.
Préconditions :
1. L'utilisateur est inscrit dans le système Robotix.
2. L'utilisateur a des robots enregistrés au système.
3. Les robots sont connectés à un résau WiFi/Bluetooth et ils sont capable de communiquer avec le système.
4. Les robots ont des capteurs pour savoir son état. Acteurs : Utilisateur
Scénario principal :
1. L'utilisateur indique les robots dont il/elle veut voir l'état.
2. Le système envoie des requètes aux robots indiqués.
3. Les robots répondent au système et renvoient leur état.
4. Le système affiche l'information reçu à l'utilisateur.
Scénarios alternatifs :
1a. L'utilisateur n'a pas de robots enregistrés.
1a.1 Le système indique à l'utilisateur qu'il faut d'abord enregistrer des robots.
2a. Un ou plusieurs robots ne peuvent pas communiquer avec le système dans le moment.
2a.1. Le système indique à l'utilisateur qu'il n'était pas capable de communiquer avec ces robots.
2a.2. Le scénario continue à l'étape 3.
3a. Un ou plusieurs robots n'ont pas de capteurs.
3a.1. Le système indique à l'utilisateur que ces robots-là n'ont pas de capteurs.
3a.2. Le scénario continue à l'étape 4.
Postconditions : L'utilisateur connaît l'état de ses robots.

Cas d'utilisation : Voir les métriques de ses robots
But : Un utilisateur souhaite voir un resume de l'acomplissement de ses robots. Distance parcourue, energie utilisee, historique de taches, performance, rendement, etc. Des données globablement utile pour planifier des ajustements.
Préconditions : L'utilisateur doit être inscrit Il doit au moins un robot
Acteurs : Utilisateur
Scénario principal : 1. L'utilisateur choisit son critère de selection, soit un robo individuel, tous les robots d'un même type, selon le fournisseur, etc
 1.1. Étape 1.1
2. L'utilisateur choisit une métrique a visionner. Une des metriques mentionée dans le but
3. L'application fait un requete d'historique a ces robots
4. Les données sont reçues et montrée en graphique ou format visuel plaisant dependant de la métrique
5. L'utilisateur peut changer le time frame des données, elle ont le time frame maximal par défault. 6. L'utilisateur peut telecharger les données en format CSV Scénarios alternatifs :
1a. Il n'y a aucun Robot qui correspond au critère de recherche
4a 1. Certains robots ne sont pas connectés. L'application propose d'ignorer les robots non connectés dans les données
4a 2. Si les robots non connectés sont ignorés, les données restent les memes mais il y a une liste dans le UI qui montre quels robots ont échoués.
4b 1. Le critère de recherche est N/A a certains robots, par example distance parcourure pour un robot immobile. L'interface demande a l'usager si il veut ces données nulles ou NULL considérée. Postconditions : L'utilisateur a des données CSV de l'information qu'il recherchait sur le groupe qu'il voulait.
